## 1. 실현 가능성 결론

**판정: "조건부 실현 가능(Conditional GO)".**
기술 파이프라인(오버레이 + 화면캡처 + 온디바이스 한글 OCR)은 **전부 표준 API로 구현 가능**하며, 이미 동일 구조의 화면 번역기 앱들이 게임 위에서 상용 동작 중이다. 우리 앱은 "번역" 자리에 "로컬 DB 조회"를 넣은 것뿐이다.

단, **코드를 짜기 전에 반드시 통과해야 하는 4개의 선검증(kill-switch)** 이 있다. 이게 막히면 완화책이 없다.

### 인식 방식 결정 (확정) — 이름 OCR 단독, 이미지 인식 불채택
- **채택: 이름 텍스트 OCR 단독.** 몇백 KB 사전 + ML Kit 한 줄로 구현되는 초경량 방식. 후보가 208종 닫힌 집합이라 fuzzy 매칭으로 오인식도 자동 교정.
- **불채택: 이미지(생김새) 인식.** 포챔스는 3D 모델이라 종별 각도·포즈·메가/폼 학습 프레임을 직접 수집·훈련해야 하고(신작이라 데이터셋 없음), 런타임도 무겁고(배터리·연산), 비슷한 종/폼 정확도도 불안. 노력 대비 실용성이 낮아 MVP·정식 모두 제외.
- **다국어 대응**: 화면 이름은 "내" 게임 언어로 렌더됨 → op.gg가 포챔스 공식 지원 9개 언어를 모두 주므로(4장 참조) 언어별 이름 테이블로 매칭. 구조적 문제 아님, 이미 구현 완료.
- **감수하는 한계**: 상대가 개명(닉네임)했거나 배틀명 표시 off면 인식 불가 → **수동 검색 fallback**으로 처리. 랭크전이 종족명을 강제하면 이 케이스는 거의 발생 안 함(K2에서 확인 대상).

### 표시명 충돌 처리 (확정) — 후보 선택 UX + 사용률 스마트 기본값
게임 화면의 배틀 표시명은 **폼 정보를 생략하고 base 종족명만 뜨는 경우가 많고, 서로 다른 포켓몬이 같은 표시명을 공유**한다(데이터 확인: name≠battleName 149건, 표시명 충돌 97건). 예: "윈디" = 윈디(불꽃) + 히스이윈디(불꽃/바위), "앱솔" = 앱솔 + 메가앱솔. **이름만으로 유일 식별 불가.**
- **매칭 구조**: OCR 문자열 → `표시명(언어별) → 후보 리스트` 인덱스로 fuzzy 매칭.
- **후보 1개 → 바로 표시. 2+ → 후보 선택 UI**: 각 후보를 **타입칩 + 사용률**과 함께 보여주고 유저가 탭해 확정. 기계가 억지 추측하지 않음.
- **사용률 스마트 정렬**: championsbattledata 사용률 높은 폼을 맨 위·기본 선택(경쟁전은 보통 한 폼만 메타 → 대개 한 번에 맞음).
- **기억**: 한 번 고르면 해당 배틀 동안 유지. 언제든 "바꾸기"로 전환.
- ⚠️ **표시명의 정확한 문자열은 K2(실기기 스크린샷)로 확정 필요.** 특히 게임이 리전폼을 "윈디"로만 쓰는지 "히스이 윈디"로 쓰는지는 실화면으로만 확인 가능. 현재는 op.gg 이름을 표시명 프록시로 사용, K2 후 문자열만 교정.

### 🚨 선검증 4종 (본개발 착수 전 필수)

| # | 검증 항목 | 방법 | 실패 시 |
|---|---|---|---|
| K1 | **게임이 `FLAG_SECURE`로 캡처를 막는가** | 실기기에 포챔스 설치 → 아무 화면녹화 앱으로 대전 화면 녹화 → 검은 화면 여부 확인 | 캡처 불가 = **앱 자체 불성립.** 우회 수단 없음(루팅 제외) |
| K2 | **상대 포켓몬 이름이 화면에 텍스트로 뜨는가** | 대전 스크린샷에서 이름 표시 확인. "Display Battle Names" 설정 기본값 확인 | 이름 미표시 = OCR 전략 붕괴. 이미지 매칭으로 전면 재설계 필요 |
| K3 | **한글 OCR 정확도/지연이 실용적인가** | 대전 스크린샷 크롭 → ML Kit 한글 PoC로 인식률·지연 실측 | 정확도 낮음 = 업스케일 전처리 + fuzzy 매칭으로 보완 시도 |
| K4 | **EULA/토너먼트 규정이 외부 보조도구를 금지하는가** | 포챔스 이용약관·랭크 규정 원문 검토 | 금지 명시 = 캐주얼 전용으로 축소 포지셔닝 |

> **가장 중요한 건 K1이다.** 닌텐도/포켓몬컴퍼니는 안티치트 의식이 강한 개발사라 `FLAG_SECURE`를 켰을 가능성을 무시할 수 없다. 이건 코드 한 줄 없이 5분이면 판정 난다. **여기부터 확인하고 시작할 것.**

## 2. 기술 스택 (확정)

iOS는 오버레이 자체가 불가하므로 크로스플랫폼 명분이 없다. 핵심 3요소가 전부 저수준 안드로이드 시스템 API라 **Kotlin 네이티브가 명확한 정답** (Flutter로 가도 이 부분들은 어차피 네이티브로 짜야 함).

| 레이어 | 선택 | 비고 |
|---|---|---|
| 언어/UI | **Kotlin + Jetpack Compose** | 오버레이 뷰도 Compose로 구성 |
| 오버레이 창 | `WindowManager` + `TYPE_APPLICATION_OVERLAY` | 터치 통과는 빈 `TOUCHABLE_INSETS_REGION` 패턴 (아래 주의) |
| 화면 캡처 | `MediaProjection` + `VirtualDisplay` + `ImageReader` | null-surface 2단계 세션 유지 |
| OCR | **ML Kit Text Recognition v2 (Korean, 번들)** | 온디바이스, 비용 0. Tesseract 대비 압도적 |
| 이름 매칭 | 로컬 포켓몬 사전 + **편집거리(Levenshtein) fuzzy 매칭** | OCR 노이즈·닉네임 대응 |
| 로컬 DB | **앱 내장 JSON** (v1), 필요 시 Room/SQLite | 레코드 ~210개, 1MB 미만이라 JSON으로 충분 |
| 백그라운드 | `foregroundServiceType="mediaProjection"` FGS | 사용자 수동 시작 |
| 데이터 갱신 | 정적 호스팅(GitHub/Cloudflare Pages) + manifest 버전 체크 | 서버비 0원 |

### 구현 시 반드시 지킬 안드로이드 최신 제약 (✅ 공식 확인)
- **Android 12+**: 불투명 오버레이로 터치를 통과시키려면 `FLAG_NOT_TOUCHABLE`이 아니라 **빈 `TOUCHABLE_INSETS_REGION`** 사용 (안 그러면 OS가 투명도를 0.8로 강제 클램프). 우리 앱은 정보 표시만 하고 터치를 안 받으므로 이 패턴이 적합.
- **Android 14**: FGS에 `foregroundServiceType="mediaProjection"` + `FOREGROUND_SERVICE_MEDIA_PROJECTION` 권한 필수. 누락 시 크래시. `MediaProjection` Intent는 1회만 사용 가능(토큰 캐싱 불가, 앱 재시작 시 재동의).
- **Android 15**: `SYSTEM_ALERT_WINDOW` 앱이 FGS를 시작하려면 **보이는 오버레이 창이 먼저 떠 있어야** 함 → "오버레이 먼저 → 서비스 시작" 순서 준수. `BOOT_COMPLETED`에서 mediaProjection FGS 시작 불가 → **자동 시작 불가, 사용자 수동 시작만.**
- **Android 15 QPR1+/16**: 캡처 중 상태바 칩 상시 노출 + 화면 잠기면 프로젝션 자동 중단 → `onStop()` 콜백 핸들링 필수.
- **세션 유지 트릭**: 대전 내내 재동의 없이 캡처하려면 null surface로 `VirtualDisplay` 유지 → 필요 시 `setSurface()`로 붙였다 뗌 (null일 땐 오버헤드 0).

## 3. 시스템 아키텍처

```
┌─────────────────────── 포챔스 서포터 앱 (Foreground Service) ───────────────────────┐
│                                                                                     │
│   [1] CaptureManager                    [2] FrameGate (트리거)                        │
│   MediaProjection → VirtualDisplay  ──▶  ROI 변화 감지(해시/diff)                      │
│   → ImageReader (다운스케일 프레임)        변화 있을 때만 다음 단계 실행 (매프레임 X)      │
│                                              │                                       │
│                                              ▼                                       │
│   [3] RoiCropper                        [4] OcrEngine                                │
│   비율 기반 이름 ROI 크롭          ──▶     ML Kit v2 (Korean) → 텍스트                  │
│   (실패 시 전체화면 OCR fallback)            │                                         │
│                                              ▼                                       │
│   [5] NameMatcher                       [6] LocalRepository                          │
│   정규화 + 편집거리 fuzzy 매칭     ──▶     내장 JSON DB 조회                            │
│   OCR "한카리아스" → dex 445                 → 타입/특성/주요기술                        │
│                                              │                                       │
│                                              ▼                                       │
│   [7] OverlayRenderer (WindowManager + Compose)                                     │
│   상대 포켓몬 정보 칩/카드를 게임 위에 렌더 (터치 통과)                                   │
│                                                                                     │
└─────────────────────────────────────────────────────────────────────────────────────┘
        ▲                                                    ▲
        │ 최초 1회                                            │ 앱 실행 시(가벼움)
   [권한 온보딩]                                        [DB 업데이트 체크]
   오버레이 권한 + MediaProjection 동의               manifest.json → 새 버전이면 다운로드
```

### 데이터 플로우 (한 사이클)
1. `CaptureManager`가 화면을 다운스케일로 캡처(상시 스트림, 저비용).
2. `FrameGate`가 이름 ROI 영역만 프레임 해시로 감시 → **변화가 있을 때만** 파이프라인 가동 (발열/전력 최소화). 대안: 1~2초 폴링.
3. `RoiCropper`가 비율 기반 ROI를 크롭. 유효 텍스트 없으면 전체화면 OCR fallback.
4. `OcrEngine`이 크롭 이미지에서 한글 텍스트 추출.
5. `NameMatcher`가 정규화 후 DB의 `name_ko_normalized`와 편집거리 매칭 → 도감번호 확정 (한두 글자 오인식/닉네임 접미사 자동 교정).
6. `LocalRepository`가 도감번호로 타입/특성/주요기술 조회.
7. `OverlayRenderer`가 오버레이 카드 갱신.

**엔드투엔드 지연 🔶 100~400ms** (사람이 읽는 정보라 밀리초 반응성 불필요, 충분).

## 4. 데이터 설계 (구현 완료 — [data/](data/))

### 4-1. 소스 & 파이프라인 (역할별 최적 소스로 확정)
```
① op.gg (pokedex + moves 페이지, 9개 언어) → 포켓몬 뼈대 + 다국어 사전
        ↓  scrape_pokedex.py → pokedex_db.json
② championsbattledata (Doubles/Singles API) → 실사용률(기술/아이템/특성/성격/파트너)
        ↓  merge_usage.py → usage_db.json (영문명으로 pokedex_db 와 slug 조인)
③ 표시명 충돌 그룹핑 (species root 기준) + 사용률 랭킹
        ↓  build_candidate_index.py → candidate_index.json
```
- **op.gg를 뼈대·다국어 소스로 채택**: 포켓덱스 페이지 RSC 페이로드에 도감번호·slug·**9개 언어 이름**·타입·특성·종족값·movepool이 완전한 JSON으로 박혀있음(스크래핑 견고, 페이지 로드 1회로 전량 확보). 별도 `moves` 페이지가 920개 기술 전체 사전을 제공해 movepool 100% 커버.
- **championsbattledata를 사용률 소스로 채택**: 공개 JSON API(`/api/battle/{Doubles|Singles}/{slug}`)로 기술/아이템/특성/성격/파트너 사용률을 %로 제공. **base stats는 안 줌**(Lv50 실능치만) + **다국어 없음**(영문 전용) → 이 두 이유로 뼈대 소스로는 부적합, 사용률 전용으로만 사용.
- ❌ pkmnchamps.com은 403 봇 차단으로 접근 불가.
- 스냅샷 방식: 포챔스 메타는 자주 안 바뀌므로 1회 수집 후 정적 파일로 보관. 소스 교체 시 각 스크립트의 어댑터만 갈아끼우면 됨(`OpggAdapter` 등).

### 4-2. 산출물 3종
| 파일 | 내용 | 규모 |
|---|---|---|
| `pokedex_db.json` | 포켓몬 317종(base+메가 75+폼), **9개 언어**(en/ja/ko/de/es/fr/it/zh-cn/zh-tw) 이름, 타입/특성/종족값/movepool, 타입·특성·기술(920개) 다국어 사전, base↔메가 상호 링크(`can_mega`/`mega_keys`/`is_mega`/`base_key`) | ~990KB |
| `usage_db.json` | 234종 × {싱글, 더블} 실사용률 — 기술/아이템/특성/성격/EV분포/파트너를 %와 함께. 기술·특성은 `pokedex_db` slug로 링크(100%), 파트너는 key로 링크(94%) | ~1.8MB |
| `candidate_index.json` | 표시명 충돌 그룹(species root) + 언어별 `정규화이름→root` 조회 테이블. 충돌 12그룹(윈디/히스이윈디 등) | ~170KB |

포챔스 지원 언어(일/영/독/스/불/이/한/중간체/중번체, 스페인어 2옵션은 종족명 동일)를 전부 커버.

### 4-3. "주요 기술" — 실사용률로 해결 (더 이상 추정 아님)
당초 우려("주요 기술을 무슨 기준으로 정하나")가 **championsbattledata 조인으로 해소됨**. 오버레이는 `usage_db`에서 해당 포맷(싱글/더블)의 사용률 상위 N개를 그대로 표시 — 손 큐레이션이나 휴리스틱 불필요. 예: 한카리아스 더블 상위 = 드래곤클로 89%·스톤샤워 84%·지진 79%·방어 73% (싱글과 다른 메타 — 포맷 분리가 실제로 중요함을 데이터로 확인).

### 4-4. 표시명 충돌 → 후보 선택 UX (확정, [5장](#5-오버레이-uiux-설계) 참조)
배틀 화면 표시명은 폼 정보를 생략하는 경우가 흔하고(예: 아르칸인 히스이폼도 화면엔 그냥 "Arcanine"), 소스마다 표기 일관성도 없음(라이츄 알로라폼은 "Alolan Raichu"로 구분 표시). → 이름만으로 유일 식별 불가하므로:
- `candidate_index.json`이 종족 루트별로 후보를 묶고, 후보들의 **모든 언어별 이름 변형을 전부 인덱싱**(어느 문자열이 실제로 뜨든 매칭되도록 세이프티넷).
- OCR 결과 → 정규화 → lookup → 후보 1개면 바로 확정, 2+면 **사용률 순 정렬 + 유저 선택 UI**(기계가 추측하지 않음).
- 메가는 이 충돌 인덱스에서 제외 — 별도의 "메가진화 토글"(4-5)로 처리.
- ⚠️ 실제 배틀 화면이 어떤 문자열을 쓰는지는 **K2(실기기 스크린샷)로 최종 확정 필요**. 현재는 op.gg 이름을 프록시로 사용.

### 4-5. 메가진화 데이터 (확정)
`pokedex_db`에 base↔메가 상호 링크 완비 — base엔 `can_mega`/`mega_keys`, 메가 레코드엔 `is_mega`/`base_key`(X/Y 2메가 보유 종 포함: 리자몽·라이츄). 오버레이 메가 토글이 이 링크로 타입/특성/종족값을 스왑; movepool·사용률은 base 그대로 재사용(메가는 기술이 안 바뀜).

### 4-6. 갱신 전략 (서버비 0원 유지)
- 산출물 3종을 정적 호스팅(GitHub/Cloudflare Pages)에 버전 태그로 배포 + `manifest.json`.
- 앱은 실행 시 manifest만 조회 → 새 버전이면 다운로드, 없으면 내장본 사용(오프라인 보장).
- 메타 변경 시 `scrape_pokedex.py` → `merge_usage.py` → `build_candidate_index.py` 순으로 재실행.

## 5. 오버레이 UI/UX 설계

### 설계 원칙
1. **게임을 가리지 않는다** — 대전 정보(HP바, 필드)를 덮지 않는 위치. 반투명 + 컴팩트.
2. **터치 통과** — 오버레이는 정보 표시 전용. 게임 조작을 절대 방해하지 않음 (빈 `TOUCHABLE_INSETS_REGION`).
3. **한눈에 읽힌다** — 타입 상성이 핵심. 색상 코딩된 타입 칩 우선.
4. **드래그로 위치 이동** — 유저가 오버레이 카드를 원하는 곳으로 옮기고 위치 저장.
5. **접힘/펼침** — 기본은 작은 칩(타입만), 탭하면 상세 카드(특성/기술)로 확장.

### 3단계 점진적 정보 공개 (탭할 때마다 펼쳐짐)
카드를 탭하면 정보량이 단계적으로 늘어난다. 각 포켓몬 카드는 독립적으로 단계를 기억.
- **1단계 · 컴팩트 칩 (기본)**: `[한카리아스] 🟤땅 🟣드래곤` — 이름 + 타입만. 가장 안 거슬림.
- **2단계 · 기본 카드**: 타입 + 특성 + 주요기술 4개 + "메가 가능" 배지.
- **3단계 · 확장 패널**: 종족값(6스탯) + **기술 전체(사용률순)** + **방어 상성(약점/반감)** + 도감번호. 경쟁전에서 가장 유용한 방어 상성을 여기서 노출.
- 더블배틀이면 상대 2마리를 세로로 나란히, 각각 독립 확장.

### 터치 모델 (구현 핵심)
오버레이는 기본이 "터치 통과"지만, 탭으로 확장하려면 **카드가 그려진 픽셀 영역만 touchable**, 나머지 빈 공간은 통과로 유지해야 한다.
- `WindowManager.LayoutParams`에서 카드 bounds만 `TOUCHABLE_INSETS_REGION`으로 지정 → 카드 위 탭은 앱이 받고, 카드 밖 탭은 게임으로 통과.
- 확장 패널은 **반투명 + 세로 스크롤 + N초 무조작 시 자동 축소**로 게임 시야를 오래 가리지 않게 함.
- 탭 순환: 칩 → 카드 → 확장 → (다시 칩). 접기 chevron으로 즉시 최소화 가능.

### 메가진화 토글 (확정)
`can_mega`인 포켓몬 카드에 **"메가진화" 버튼** 노출 → 누르면 메가 상태 데이터(바뀐 **타입/특성/오른 종족값**)로 스왑. 데이터 근거: `pokedex_db`에 base↔메가 상호 링크(`can_mega`/`mega_keys`/`is_mega`/`base_key`) 완비, 73 base 링크됨.
- **X/Y 2개 메가(리자몽·라이츄)**는 세그먼트 토글([메가 X][메가 Y])로 전환.
- **기술은 스왑 안 함** — 메가는 movepool이 base와 동일하므로 기술/사용률은 base 데이터 재사용. 타입/특성/종족값만 메가 값으로 교체.
- 예: 메가리자몽X는 불꽃/비행 → **불꽃/드래곤**(타입 변화), 특성 강철날개, 합 534→634.

### 상태별 UX
| 상태 | 오버레이 표시 |
|---|---|
| 인식 성공 | 정상 카드 |
| OCR 미매칭 | "인식 실패 🔍" + 탭하면 수동 검색 |
| 이름 미표시(닉네임/설정 off) | "배틀명 표시를 켜주세요" 1회 안내 |
| 캡처 중단(잠금 등) | 오버레이 최소화 + 재시작 버튼 |

> 실제 목업은 별도 시각 자료로 제공(오버레이가 게임 화면 위에 얹힌 모습).

## 6. 권한 & 온보딩 흐름 (1회)

```
1. 앱 최초 실행 → 온보딩: "이 앱은 화면에 이미 보이는 정보만 표시합니다" 고지
2. "다른 앱 위에 표시" 권한(SYSTEM_ALERT_WINDOW) → 시스템 설정으로 이동해 허용
3. 게임 설정 안내: "포챔스 → 설정 → Display Battle Names ON" 안내(K2 대응)
4. "시작" 버튼 → 오버레이 먼저 표시 → FGS 시작 → MediaProjection 시스템 동의
5. (선택) ML Kit 한글 모델 번들이면 다운로드 불필요
6. 상태바에 "캡처 중" 칩 상시 노출 (Android 정책, 투명성 확보)
```
- **자동 시작 불가** (Android 15 정책) → 매 대전 세션마다 사용자가 직접 "시작". 안티치트 관점에서도 바람직.
