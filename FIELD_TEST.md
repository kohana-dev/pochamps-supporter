> ## ⚡ P24 갱신 — 오버레이 기본 터치 통과(NOT_TOUCHABLE) + 상호작용 토글 핸들 (2026-07-05, v0.1.10)
> 실사용자 리포트(게임 터치 안 됨 / 화면 과다 점유 / 상시 진단 스트립)를 근본 수정했다.
> **메인 정보/컨트롤 창을 기본 `FLAG_NOT_TOUCHABLE`(통과)로 두고**, 아주 작은 별도 **상시 touchable 핸들 창**을
> 탭하면 잠깐 상호작용 모드(메인 창 touchable)로 바뀌고 6s 무조작이면 자동 통과 복귀한다.
> 진단 스트립은 진단 ON + DiagState 있을 때만, 헬스는 문제일 때만 표시(정상=카드만).
>
> ### 실기기 확인 항목 (P24 — 터치 통과)
> 1. **게임 터치 통과(핵심)**: 캡처 시작 후 **카드/컨트롤이 보이는 영역을 탭했을 때 그 터치가 아래 게임으로 통과**하는지.
>    - 검증법: `adb shell dumpsys window windows` 로 **메인 카드 창에 `NOT_TOUCHABLE` 플래그가 있는지** 확인
>      (핸들 창에는 없어야 함). 또는 오버레이 아래에 탭 로그가 남는 테스트 액티비티를 두고 카드 위를 탭해
>      **아래 앱이 탭을 받는지** 확인. (에뮬 실측: 카드 위 탭이 아래 "Stop" 버튼으로 통과되어 캡처가 멈춤 → 통과 확정.)
> 2. **핸들 탭 → 상호작용 → 자동 복귀**: 구석 핸들(🔒) 탭 → ✋(조작중)로 바뀌고 메인 창이 touchable 로 전환 →
>    카드 펼치기/검색/보정/후보선택 동작 → 6s 무조작 또는 재탭 시 🔒(통과중)로 복귀(다시 게임 터치 통과).
> 3. **핸들 길게 누르기 → 최소화(👁)/복원**: 카드·컨트롤 전체를 접었다 펴는지(위치는 저장·유지).
> 4. **정상 인식 평상시 화면 = 컨트롤/핸들 + 포켓몬 카드만**(진단 스트립·헬스"정상" 라인 없음).
> 5. **진단 ON 시에만** 컨트롤 바 "Diag" 하이라이트 + (실캡처 DiagState 존재 시) 스트립 표시.
> - 스크린샷: `field_samples/p24/`.

>
> ### 실기기 확인 항목 (P22 — 더블 ROI)
> 1. **실 더블배틀에서 두 이름표가 각 밴드 안에 온전히 잡히는지**: 좌 밴드에 좌 포켓몬 이름 전체, 우 밴드에 우 포켓몬
>    이름 전체(파이어로처럼 긴 이름의 **왼쪽 글자 클리핑 없음**)가 들어오는지 오버레이 진단 스트립으로 확인.
> 2. **우 밴드가 커맨드 버튼/HP 위젯을 물지 않는지**(right=0.940 로 조임 — 버튼 텍스트 오탐 0 확인).
> 3. 기기별 노치/펀치홀·종횡비 편차 시 보정 UI(`[⃞]`)로 미세조정 → 형식별 오버라이드로 영속(P20).
> 4. 인식 실패 시 카드 유지(안전 열화) + `[🔍]` 수동 검색 진입(P21) 동작 확인.

> ## ⚡ P20 갱신 — 싱글/더블 배틀 형식 토글 (v0.1.4, 2026-07-05)
> 기존엔 앱이 **항상 더블로 하드코딩**돼 싱글 배틀이 작동 불가였다(ROI 어긋남 + 더블 메타 오표시). 이제 **수동 토글**로 형식(ROI+사용률)을 함께 전환한다.
> - **오버레이 빠른 토글**: 오버레이 상단 **[싱글][더블] 세그먼트**로 대전마다 즉시 전환. 세그먼트만 터치, 그 밖은 게임으로 통과(창=카드 전략 보존).
>   - 토글 즉시: ROI 밴드 수(싱글 1 / 더블 2)와 사용률(싱글 vs 더블 메타)이 다음 프레임부터 갱신. 더블→싱글 시 남는 2번째 카드 정리, Decider/FrameGate 리셋(고착 방지).
> - **설정 토글**: MainActivity 설정 → **"배틀 형식"** 싱글/더블 칩. (다음 실행부터 반영; 실행 중 변경은 오버레이 빠른 토글이 즉시.)
> - **ROI 보정 형식별 분리(P14 연동)**: 싱글/더블은 밴드 수가 달라 오버라이드를 **형식별로 따로** 저장한다. **"이름 영역 보정"·"ROI 리셋"은 현재 형식의 것만** 편집/리셋(다른 형식 보정 보존). → 싱글 보정과 더블 보정을 각각 잡아 두면 토글마다 자동 적용.

> ## ⚡ P18 갱신 — 모든 카드에서 수동 지정(🔍) + 강제 재인식(↻) (v0.1.5, 2026-07-05)
> 앱이 **자신 있게 틀린 포켓몬**을 띄우면 이전엔 고칠 수 없었다(수동 검색이 "인식 실패" 카드에서만 열림). 이제 **어떤 카드에서든** 교정·재인식할 수 있다. **회복 사다리**: 자동 재스캔 → ↻ 강제 재인식 → 🔍 수동 지정 → ROI 보정(P14) → 진단(P14).
> - **🔍 수동 지정(모든 카드)**: 카드 그립 바 우측 🔍 탭 → 이름 검색 시트(P16 가로 flyout·IME 포커스 토글 재사용) → 선택 시 그 슬롯 **핀**(파이프라인이 안 덮어씀). 정상적으로 잘못 매칭된 카드도 **전혀 다른 포켓몬으로 교정** 가능. 핀 상태는 `📌 고정 해제` 배지로 표시, 탭하면 자동 인식 복귀.
> - **↻ 강제 재인식(정상+실패 카드)**: 카드 그립 바 우측 ↻ 탭 → "지금 이 슬롯 다시 읽어". Decider 슬롯 상태 리셋(오인식 고착 탈출) + FrameGate 게이트 **1회 우회**(정지 화면이라도 다음 프레임 즉시 OCR). 핀 상태면 ↻ 가 **핀을 풀고 재인식**. 더블이면 카드별 개별 ↻.
> - 에뮬 실측(데모): (a) 데모 **Garchomp** 카드에서 🔍 → "arcanine" 검색 → 선택 → **카드가 Arcanine(Fire)로 교체 + 핀 배지**(`field_samples/p18/a1~a4`). (b) ↻ 탭 → `forceRescan` 콜백 발화(logcat, 데모는 파이프라인 no-op). (c) 핀 해제 → 배지 사라짐(`c1`). 유닛 +8(총 **201/201**). release **versionCode 7 / 0.1.5**, 데이터 URL 불변.
> - **실기기 잔여(정직)**: 실 대전에서 ↻ 가 정지 화면 오인식 고착을 실제로 깨는지 최종 확인(에뮬 데모는 파이프라인 no-op — 로직은 JVM 테스트로 보증). 🔍 수동 지정은 실캡처도 데모와 동일 경로(`pipeline.pinSlot`).

## (b) K1 — FLAG_SECURE 캡처 차단 검증 ★최우선(5분)★

**목적**: 포챔스가 화면 캡처를 막는가(`FLAG_SECURE`). 막으면 이 앱은 성립 불가(우회 불가, 루팅 제외).
**이 앱 없이도 판정 가능** — 코드 한 줄 없이 5분.

절차:
1. 포챔스를 실행해 **실제 대전 화면**(상대 포켓몬이 보이는 화면)까지 진입.
2. 기기 기본 **화면 녹화**(빠른 설정 타일) 또는 아무 스크린샷을 대전 화면에서 찍는다.
3. 녹화 영상/스크린샷을 열어 확인:
   - **대전 화면이 정상적으로 보이면 → K1 PASS.** 본 앱 진행 가치 있음.
   - **검은 화면/캡처 차단 문구가 뜨면 → K1 FAIL.** `FLAG_SECURE` 사용 = **프로젝트 중단**.
4. (교차 확인) 본 앱으로도 검증: 앱 설치 → "시작(화면 캡처)" → 대전 화면에서
   ```bash
   adb logcat -s CaptureManager RecognitionPipeline
   ```
   프레임이 계속 검게(전부 동일/검정) 들어오면 FLAG_SECURE 의심.

> **P17 자동화(v0.1.3~)**: 이제 **앱이 스스로 K1 을 감지·고지**한다(수동 판정 불필요, 확인만).
> - 대전 화면에서 캡처가 **검은/균일 저휘도** 프레임을 2.5s 연속 주면 → 오버레이에
>   **"화면 캡처가 차단됨(검은 화면)"** 경고 카드를 자동 표시 = **K1 FAIL 자동 감지**.
> - 프레임이 4s 동안 아예 안 들어오면 → **"화면 프레임 없음 + 재시작"** 카드.
> - 정상(밝은 프레임 갱신)이면 안내 없음 + (진단 모드 시) 스트립에 "캡처: 정상".
> - 따라서 실기기 K1 확인은 "**대전 화면에서 검은-화면 경고 카드가 뜨는가?**" 한 번으로 끝난다.
>   에뮬 실측(AVD)으로 검은 영상→경고 카드, 정상 영상→무경고·정상 복귀를 이미 확인
>   (`field_samples/p17/`). 실기기 정식판에서 최종 확인만 남음.

> 판정이 애매하면(일부 화면만 검정 등) 대전 중 여러 화면에서 반복 확인. 로비만 캡처되고
> 대전 화면만 검정이면 그것도 FAIL 취급.

## (c) K2 — 표시명 확정 + ROI 좌표 보정

**목적**: (1) 배틀 화면에 뜨는 상대 이름 **문자열**이 우리 DB 표시명과 맞는지, (2) 이름 텍스트의
**화면상 위치(ROI)** 를 기기 해상도에 맞게 보정.

현재 ROI 기본값은 추정치다(`RoiConfig.DEFAULT_LANDSCAPE_DOUBLES`, 가로 더블배틀 상단 좌/우 2곳).
실좌표는 **앱 내 보정 화면**으로 맞춘다(P14).

### 절차 (권장 — 앱 내 보정, P14)
1. 앱 실행 → 설정 → **"이름 영역 보정 (화면에서 직접)"** 탭(오버레이 권한 필요).
2. 게임 화면 위에 반투명 편집기가 뜬다. 현재 밴드가 색 사각형(싱글 1개/더블 2개)으로 표시된다.
3. **싱글/더블 탭**으로 배틀 형식을 고르고, 각 밴드를:
   - **안쪽을 드래그 → 이동**, **모서리(핸들)를 드래그 → 크기 조절** 하여 **상대 이름표 텍스트에 정확히 맞춘다**.
   - 어긋났으면 **"초기화"** 로 기본값 복원 후 다시.
4. **"저장"** → 즉시 파이프라인이 다음 프레임부터 새 ROI 를 쓴다(재시작 불필요). "닫기"로 편집기 종료.
5. **진단 모드**(설정 토글)를 함께 켜두면, 보정 직후 스트립에서 인식 여부(root/editDistance/빈텍스트)를 바로 확인 가능.

> 팁: 진단 스트립이 **"OCR:빈텍스트"** 면 밴드가 이름표를 벗어났거나 글자가 잘린 것(밴드를 이동/확대).
> **"미매칭 \"...\""** 면 OCR 은 됐는데 DB 표시명과 다른 것(리전폼/닉네임 → 수동 검색 or DB 갱신).

### 절차 (고급 대안 — adb 스크린샷 좌표 계산)
> P14 의 인앱 보정으로 대부분 끝난다. 아래는 픽셀 단위 정밀 측정이 필요할 때의 대안.
1. 대전 화면 스크린샷을 여러 장 찍는다(더블배틀, 상대 2마리 이름이 보이게).
   ```bash
   adb exec-out screencap -p > shot.png
   ```
2. 스크린샷을 이미지 편집기로 열어 **상대 이름 텍스트를 감싸는 사각형**의 픽셀 좌표를 잰다.
   각 ROI(좌/우 이름)마다: `left, top, right, bottom` (픽셀).
3. 화면 전체 크기(`W = 가로픽셀`, `H = 세로픽셀`)로 **비율(0~1)** 로 환산:
   ```
   left/W, top/H, right/W, bottom/H
   ```
4. 이 비율값으로 ROI 를 보정한다:
   - **코드 수정**: `android/.../capture/RoiConfig.kt` 의 `DEFAULT_LANDSCAPE_DOUBLES`/`SINGLE` 를 교체 → 재빌드.
   - **런타임 오버라이드**(재빌드 없이): `roi_prefs.xml` 에 `roi_config` = `"l,t,r,b;l,t,r,b"`(더블 2밴드) 또는 `"l,t,r,b"`(싱글) 직렬화 저장.
     (adb `run-as` 로 SharedPreferences 주입, 또는 위 인앱 보정 화면의 "저장".)
5. 재실행 → 이름 위에 카드가 정확히 뜨는지 확인. 어긋나면 반복.

### 표시명 문자열 확정
- 스크린샷의 **정확한 이름 텍스트**(예: "윈디" vs "히스이 윈디", "Arcanine" vs "Hisuian Arcanine")를
  기록한다. 리전폼/폼 변화가 base 이름으로만 뜨는지 확인.
- 현재 DB 표시명은 op.gg 이름을 프록시로 사용 중 → 실화면과 다르면
  `data/candidate_index.json` 의 lookup 키를 실제 문자열로 교정(파이프라인은 [4-1] 재실행).
- **닉네임 설정 시** 종족명 대신 닉네임이 뜨는지도 확인(뜨면 수동 검색 fallback 대상).

### 방법 A — `OcrFieldTest` 계측 하네스(권장, 자동 표 출력)
`field_samples/` 실배틀 5장을 실 파이프라인에 통과시키는 하네스. 에뮬/실기기 연결 후:
```bash
cd android
./gradlew :app:connectedDebugAndroidTest \
  -Pandroid.testInstrumentationRunnerArguments.class=com.pochamps.supporter.OcrFieldTest
adb logcat -s OcrFieldTest      # 이미지별 인식라인/매칭root/지연ms/판정 표
```
- 테스트 3개: `k3_ocr_실측_field_samples`(기본 경로), `k3_ocr_전처리_grayscale_contrast`(전처리 비교),
  `k3_diag_hippowdon2_roi_variants`(ROI 진단 격자). 특정 1개만: 위 `class=...#메서드명`.
- **P9 실측 결론**: 기본(전처리 없음) 경로가 7/7. GRAYSCALE_CONTRAST 는 이 샘플에선 개선 없음(실기기 저대비 대비 옵션).
  hippowdon2 오인식은 ROI 하단 클리핑이 원인이라 **ROI bottom 확장**으로 해결(전처리 아님).
정확도가 낮으면 아래 "튜닝 순서"대로 조정. (샘플이 부족하면 실배틀 스크린샷을 assets 에 추가.)

### 방법 B — 앱 내 진단 모드(권장, adb 불필요 — P14)
설정 → **"진단 모드" 토글 ON** → 캡처 시작 → 오버레이 카드 밑 **진단 스트립**을 본다:
- 슬롯별 한 줄: 매칭이면 `S0 <root> d<editDistance>`(초록), 빈텍스트면 `S0 OCR:빈텍스트`(빨강), 미매칭이면 `S0 미매칭 "<원문>"`(주황).
- 하단에 **마지막 인식 경과**("방금/n초 전 인식")와 **OCR 회/s**(발열/스로틀 판단).
- 인식률/원인 판단이 폰 화면에서 바로 된다 → 아래 logcat 은 더 깊은 추적이 필요할 때만.

### 방법 C — 실배틀 logcat (고급)

### logcat 태그
debug APK 로 실행 후:
```bash
adb logcat -s CaptureService CaptureManager RecognitionPipeline OcrEngine
```
관찰 포인트:
- `CaptureService`: "파이프라인 시작 완료" → 세션/파이프라인 기동.
- `CaptureManager`: 프레임 수신 흐름(다운스케일/스로틀).
- `RecognitionPipeline`: "프레임 처리 실패" 경고가 잦으면 크롭/OCR 문제.
- `OcrEngine`: 추출된 라인.

### 실측 방법
1. 대전 화면에서 상대 포켓몬 여러 종을 노출시키며 **인식된 이름**과 **정답**을 대조 → 정확도(%).
2. OCR 호출당 지연(ms)을 로그 타임스탬프로 집계. **목표 100~400ms**(DESIGN.md 3장).
3. 정확도가 낮으면 튜닝 순서:
   - **ROI 여백(특히 하단)을 넓혀 글자 잘림 방지** — P9 실측에서 가장 효과 큰 튜닝(hippowdon2 클리핑 사례).
   - `RoiCropper` 업스케일 배율 상향(2x → 3x) — 작은 폰트 개선.
   - `OcrEngine(..., Preprocess.GRAYSCALE_CONTRAST)` — 저대비/이탤릭 프레임 대비(P9 샘플엔 무효였으나 실기기 프레임 대비).
   - `FrameGate` `diffThreshold`/최소 인터벌 조정(과트리거/미트리거).
   - 언어 설정이 게임 언어와 일치하는지(설정 → 게임 언어) — recognizer 스크립트가 바뀐다.

### 발열/전력 확인
- 이름이 안 바뀌는 동안 OCR 이 **안 돌아야** 한다(`FrameGate` 가 변화 없을 때 트리거 안 함).
  로그에 유휴 시 OCR 호출이 없으면 정상.

## (f) 알려진 제약 / 트러블슈팅

### IME 포커스 (P5 수정 반영)
- 오버레이 창은 평소 `FLAG_NOT_FOCUSABLE`(게임 키 포커스 보존). 수동 검색 시트가 열리면
  이 플래그를 **잠시 제거**해 키보드 입력을 받고, 닫으면 복원한다(`OverlayRenderer.setFocusable`).
- **확인 포인트**: 검색 시트에서 소프트 키보드가 뜨고 글자가 입력되는가. 안 되면:
  - 로그로 `updateViewLayout` 예외 여부 확인.
  - 일부 기기/런처는 오버레이 창 IME 를 제약할 수 있음 → 그 경우 검색을 MainActivity(일반 창)로
    분리하는 대안 고려(현재는 오버레이 내 입력이 기본).

### Android 14+ 재동의
- MediaProjection 토큰은 **1회성**(캐싱 불가). 앱 재시작/캡처 중단(화면잠금) 후 **다시 동의**해야 한다.
- 그래서 "캡처 중단됨" 카드의 "재시작" 은 앱을 다시 열어 동의 다이얼로그를 새로 띄운다(정상 동작).
  - 재시작은 기존 토큰을 재사용하지 **않는다**(재사용 시 Android 14+ 에서 SecurityException).
    `restartCapture()` 가 서비스를 `stopSelf()` 하고 MainActivity 를 앞으로 가져오면, 액티비티가
    `ON_RESUME` 에서 "시작" 버튼을 복원해(P7 수정) 유저가 `createScreenCaptureIntent` 로 재동의한다.
    화면잠금 자동중단·상태바 칩 중단·유저 중단 모든 경로가 이 재동의 흐름을 탄다.

### FGS specialUse — Play 심사 주의(배포 단계)
- 데모 경로(MediaProjection 토큰 없이 오버레이 UI 만 검증)는 `FOREGROUND_SERVICE_SPECIAL_USE` 타입으로
  FGS 를 시작한다(매니페스트 `foregroundServiceType="mediaProjection|specialUse"` +
  `PROPERTY_SPECIAL_USE_FGS_SUBTYPE="overlay_ui_demo_without_capture"`).
- **Google Play 정책상 `specialUse` FGS 는 콘솔에 사용 사유 선언·심사가 필요**하다. 배포 시 앱 콘텐츠 →
  포그라운드 서비스 항목에 "MediaProjection 동의 없이 오버레이 UI 를 미리보기(데모)하기 위한 단기 FGS" 취지로
  기재해야 한다. 심사 통과가 어려우면 데모 경로를 릴리즈 빌드에서 제외하고 `specialUse` 타입/권한/속성을
  매니페스트에서 제거하는 것도 대안(실캡처 경로는 `mediaProjection` 타입만 사용하므로 무관).

### Android 15 순서
- SYSTEM_ALERT_WINDOW 앱이 mediaProjection FGS 를 시작하려면 **보이는 오버레이가 먼저** 떠 있어야 함.
  서비스가 `onStartCommand` 에서 "오버레이 show() → startForeground(MEDIA_PROJECTION)" 순으로 처리한다.
  `ForegroundServiceStartNotAllowedException` 이 뜨면 이 순서/오버레이 권한을 의심.
- BOOT_COMPLETED 자동 시작 불가 → **매 세션 사용자 수동 시작**.

### 상태바 "캡처 중" 알림
- POST_NOTIFICATIONS(Android 13+) 거부 시 알림이 안 뜬다(FGS 자체는 동작). 온보딩에서 요청함.
- Android 15 QPR1+/16 은 캡처 중 상태바 칩을 상시 노출(정책).

### 터치/드래그
- 시트/확장으로 창이 커진 상태에서도 창 밖 게임 터치가 통과하는지, 화면경계 클램프가 정상인지 확인.

### Image→Bitmap 정합
- 일부 GPU 는 `rowStride` 패딩이 있어 이미지가 우측으로 밀릴 수 있다. 패딩 크롭이 되는지
  (카드에 엉뚱한 이름/깨진 크롭) 확인. 문제 시 `CaptureManager` 의 패딩 처리 로그 확인.

### 약관/안티치트
- 경쟁 VGC 타이틀이므로 **"화면에 공개된 정보 표시"에만** 한정. 메모리 훅/자동입력 없음.
- **K4 판정(P10 완료)**: 아래 (g)절 참조. **공인 대회는 명백 금지, 캐주얼도 회색지대** → 앱은 **캐주얼 전용으로 포지셔닝**하고 온보딩에 경고 고지.
